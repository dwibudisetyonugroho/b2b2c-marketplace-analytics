# Step 3: Data Requirements
## The Relational Matrix Mapping for B2B2C Analytics

### 3.1 Strategic Overview
In the previous steps, we defined *what* business problems we are solving (Business Understanding) and *how* we will solve them analytically (Analytic Approach). Now, we must define exactly *which* data elements are required to execute those solutions.

As the lead analyst, I have audited the Olist dataset metadata and identified five core tables that form the backbone of **The B2B2C Marketplace Growth & Retention Framework**. I am not just collecting "all the data"; I am selectively extracting specific fields that map directly to our Key Performance Indicators (KPIs).

### 3.2 Phase 1 Data Requirements: B2B Merchant Acquisition
To calculate **Funnel Velocity**, **Conversion Rates**, and **Revenue Yield per Lead**, I need to bridge three distinct operational stages: Lead Generation → Sales Closure → Post-Sale Revenue.

#### A. Top-of-Funnel: `leads_qualified.csv`
This table represents the initial marketing touchpoint.
| Required Column | Data Type | Business Justification |
| :--- | :--- | :--- |
| `mql_id` | String (UUID) | **Primary Key.** Essential for joining qualified leads to their final closed status. |
| `first_contact_date` | Date/String | **Temporal Anchor.** Used to calculate the start of the sales cycle for "Days to Close." |
| `origin` | String | **Segmentation Key.** This is our primary dimension for analyzing which marketing channel (e.g., 'social', 'organic') performs best. |

#### B. Mid-Funnel: `leads_closed.csv`
This table represents the successful conversion of a lead into an active seller.
| Required Column | Data Type | Business Justification |
| :--- | :--- | :--- |
| `mql_id` | String (UUID) | **Join Key.** Links the closed deal back to the original marketing source. |
| `seller_id` | String (UUID) | **Entity Link.** Connects the sales outcome to the specific merchant who will generate future GMV. |
| `won_date` | DateTime/String | **Temporal Anchor.** Used to calculate the end of the sales cycle for "Days to Close." |

#### C. Bottom-of-Funnel: `order_items.csv`
This table provides the financial proof of a seller's value.
| Required Column | Data Type | Business Justification |
| :--- | :--- | :--- |
| `seller_id` | String (UUID) | **Aggregation Key.** Allows me to sum up all sales generated by a specific acquired seller. |
| `price` | Numeric | **Value Metric.** The core component of Gross Merchandise Value (GMV). Used to calculate "Revenue Yield per MQL." |

### 3.3 Phase 2 Data Requirements: B2C Customer Retention
To build a **Cohort Retention Matrix**, I must resolve customer identity across multiple transactions. The core challenge is that `customer_id` changes per order, so I must rely on `customer_unique_id`.

#### A. Entity Resolution: `customers.csv`
This table allows me to track the same human being over time.
| Required Column | Data Type | Business Justification |
| :--- | :--- | :--- |
| `customer_id` | String (UUID) | **Transaction Link.** Joins individual orders to the master customer record. |
| `customer_unique_id` | String (UUID) | **Longitudinal Key.** The single most important column for retention analysis. It remains static regardless of how many times a user buys. |

#### B. Transactional History: `orders.csv`
This table provides the timeline of purchasing behavior.
| Required Column | Data Type | Business Justification |
| :--- | :--- | :--- |
| `order_id` | String (UUID) | **Granularity Key.** Ensures we are counting unique transactions, not just line items. |
| `customer_id` | String (UUID) | **Join Key.** Links the order to the `customers` table to retrieve the `customer_unique_id`. |
| `order_status` | String | **Quality Filter.** Critical for filtering out cancelled or failed orders. We only analyze `'delivered'` orders to measure true satisfaction and retention. |
| `order_purchase_timestamp` | DateTime/String | **Cohort Anchor.** Used to determine the "Cohort Month" (when the user first arrived) and the "Return Month" (when they came back). |

### 3.4 Data Integrity & Constraint Checklist
Before proceeding to Data Collection, I have established the following technical constraints to ensure our SQL models remain robust:

1.  **Temporal Alignment:** All date fields (`first_contact_date`, `won_date`, `order_purchase_timestamp`) must be convertible to standard SQL `DATE` or `DATETIME2` formats. Any non-standard string formatting will require cleaning during the Data Preparation phase.
2.  **Key Uniqueness:**
    *   `mql_id` must be unique in `leads_qualified` to prevent double-counting marketing spend.
    *   `customer_unique_id` must be stable across the entire dataset to prevent fragmentation of cohort sizes.
3.  **Null Handling Strategy:**
    *   For `origin` in `leads_qualified`: If NULLs exist, they must be categorized as 'Unknown' rather than dropped, to preserve total lead volume accuracy.
    *   For `order_status` in `orders`: We will strictly exclude any record where status is not `'delivered'` to avoid skewing retention metrics with failed logistics attempts.

### 3.5 Excluded Data Fields
To maintain a lean and focused analytical pipeline, I am explicitly excluding the following fields from this specific project scope:
*   `geolocation.csv`: While useful for spatial analysis, it is not required for funnel velocity or cohort retention calculations.
*   `product_category_name_translation.csv`: Product-level details are aggregated at the `seller_id` level for B2B yield; granular product categories are out of scope for this high-level marketplace health check.
*   `order_reviews.csv`: Sentiment analysis is a separate project domain. We are focusing purely on behavioral retention (did they buy again?) rather than attitudinal retention (did they like it?).
